La qualité logicielle

 Une crise du logiciel est survenue dans les années 60-70.
  • Des délais de livraison non tenus
  • Des dépassements de 50% à 70% des coûts prévisionnels
  • Des anomalies techniques
  • Des décalages avec les besoins du client
  • Des difficultés à réaliser des évolutions
  • Des difficultés à réutiliser le logiciel dans d'autres contextes de développement
  • ...

Qu'est-ce que c'est ?

 La qualité logicielle regroupe tous les indicateurs et techniques permettant d'évaluer et d'améliorer la qualité d'un logiciel ou des processus.

Les idées reçues

 La qualité logicielle véhicule un certain nombre d'idées reçues...
  • Tester coûte cher !
  • Tester est une perte de temps !
  • Si ça ne marche pas, soit c'est le client qui s'est mal exprimé, soit c'est l'utilisateur qui ne sait pas se servir du logiciel !
  • Une fois livré, les problèmes ne sont plus de ma responsabilité...
  • On ne peut tester un logiciel que lorsqu'il est codé !
  • Ce sont les testeurs qui doivent tester !
  • Les tests doivent être automatiques !
  • Plus il y a de tests mieux c'est !
  • Tester permet de trouver tous les bugs !
  • Si tous mes tests passent, mon logiciel est de qualité !

Fiabilité

 La fiabilité d'un code est sa capacité à s'exécuter correctement dans les cas nominaux comme dans les cas aux limites, et à produire le résultat escompté.
  • Tests unitaires
  • Tests d'intégration
  • Métriques
 C'est le niveau le plus simple à mettre en place.

Les tests unitaires

 Les tests unitaires ont pour but de tester une partie ou une portion de code de manière unitaire et indépendante.
  • Un bon test unitaire doit être...
  • unitaire
  • indépendant
  • simple
  • reproductible
  • rapide
  • clair
 Idéalement, ils sont codés avant l'implémentation de la fonctionnalité, on appelle cela le TDD pour Test Driven Development.

JUnit

 JUnit est un framework java permettant de réaliser des tests unitaires.
  • Classe et méthodes de test
  • Vérifications
  • Tests paramétrés
  • Ressources de test
  • Tests avec Spring
  • Initialisation et Réinitialisation
  • Mock
 Nous allons voir JUnit 4 encore très utilisé aujourd'hui et JUnit 5 qui propose des fonctionnalités plus poussées.

Classe et méthodes de test

 Les classes contenant les tests unitaires sont généralement placées dans le répertoire src/test/java. Ils sont nommés NomClasseTest ou NomClasse correspond à la classe testée et sont situées dans le même package. 
public class VehiculeTest {

  @Test //Junit 4 : org.junit.Test, Junit 5 : org.junit.jupiter.api.Test
  public void testCheckBadImmatriculation(){
      //Given = Initialisation des données d'entrée
      String immatriculation = "XXXXXX";

      //When = Exécution de la méthode à tester
      Boolean immatOk = Vehicule.checkImmatriculation(immatriculation);

      //Then = Vérifications de ce qu'a fait la méthode
      Assertions.assertThat(immatOk).isFalse();
}
 On nomme généralement le nom de la méthode de test de manière à comprendre ce que vérifie le test. Exemple : testCheckBadImmatriculation ou shouldBadImmatriculation.

Vérifications

 Pour vérifier le comportement de la méthode testée, on utilise des assertions. JUnit propose des assertions de base mais voyons les assertions proposées par AssertJ qui sont plus complètes et lisibles car sous la forme assertThat(objectAVerifier).testAEffectuer(). 
public class VehiculeTest {
  @Test
  public void testCheckBadImmatriculation(){
    //Given, When
    //Then (org.assertj.core.api.Assertions)
    Assertions.assertThat(booleen).isFalse(); //isTrue
    Assertions.assertThat(booleen).isNull(); //isNotNull
    Assertions.assertThat(o).isEqualTo(o2); //isNotEqualTo
    Assertions.assertThat(s).contains("t"); //doesNotContain
    Assertions.assertThat(s).startsWith("t"); //endWith
    Assertions.assertThat(s).isEmpty(); //isNotEmpty
    Assertions.assertThat(n).isPositive(); //isNegative
    Assertions.assertThat(n).isGreaterThan(5); //isLessThan
    Assertions.assertThat(n).isGreaterThanOrEqualTo(5);
    Assertions.assertThat(list).hasSize(3);
    ...
}

public class VehiculeTest {
  @Test
  public void testCheckBadImmatriculation(){
    //Given, When
    //Then org.junit.jupiter.api.Assertions;
    Assertions.assertFalse(booleen); //assertTrue
    Assertions.assertNull(booleen); //assertNotNull
    Assertions.assertEquals(expected, actual); //assertNotEquals
    Assertions.assertSame(expected, actual); //assertNotSame
    Assertions.assertArrayEquals(expected, actual);
    ...
}
 Pour connaître toutes les méthodes disponibles, consulter la documentation de AssertJ ou de Junit ou utiliser l'autocomplétion de votre IDE.

Tests paramétrés (Junit 4)

 Lorsque l'on veut utiliser plusieurs jeux de données pour un même test, on peut utiliser le paramétrage de test. 
@RunWith(value = Parameterized.class)//org.junit.runners.Parameterized
public class VehiculeTest {
  @Parameter(value = 0)//org.junit.runners.Parameterized.Parameter
  public String immat;
  @Parameter(value = 1)
  public Boolean result;
  //org.junit.runners.Parameterized.Parameters
  @Parameters(name = "immat {0} est valide : {1}")
  public static Collection<Object[]> data() {
      return Arrays.asList(new Object[][]{
        {"XXXXX", false}, {"AA-123-BB", true}
      });
  }
  @Test
  public void testCheckBadImmatriculation(){
      //Given, When, Then
      Assertions.assertThat(Vehicule.checkImmat(immat)).isEqualTo(result);
  }
 Cela évite de dupliquer inutilement du code et facilite l'ajout de nouveaux tests. L'inconvénient est qu'on doit définir ces paramètres au niveau de la classe de test et pas indépendamment chaque méthode de test.

Tests paramétrés (Junit 5)

 Voici la syntaxe plus simple avec Junit 5. 
@ParameterizedTest(name = "immat {0} est valide : {1}")
@CsvSource({
        "'XXXXX', false",
        "'AA-123-BB', true"
})
void testCheckBadImmatriculation(String immat, Boolean result) {
    //Given, When, Then
    Assertions.assertThat(Vehicule.checkImmat(immat)).isEqualTo(result);
}
 D'autres syntaxes possibles sont expliquées dans la documentation.

Ressources de test

 Il est fréquent de vouloir paramétrer notre application différement lorsque l'on est dans les tests que lorsqu'elle est lancée normalement. On peut créer un fichier application.properties dans le répertoire src/test/resources en y mettant les propriétés que l'on souhaite redéfinir. 
//src/test/resources/application.properties

spring.datasource.driver-class-name=org.h2.Driver
spring.datasource.url=jdbc:h2:mem:db;DB_CLOSE_DELAY=-1
spring.datasource.username=sa
spring.datasource.password=sa
 Par exemple, on utilisera une BDD MySQL pour l'application et une BDD en mémoire éphémère pour les tests.

Tests avec Spring

 Lorsqu'on veut tester des classes gérées par Spring, il est nécessaire que le contexte Spring soit lancé en même temps que le test (pour que les annotations @Autowired fonctionnent). 
@RunWith(SpringRunner.class) //Junit 4
@ExtendWith(SpringExtension.class) //Junit 5 
@DataJpaTest // ou @SpringBootTest
public class VehiculeRepositoryTest {

    @Autowired
    VehiculeRepository vehiculeRepository;
    @Test
    public void testFindByImmat(){
        //Given
        Vehicule v = vehiculeRepository.save(new Vehicule("AA-123-BB"));
        //When
        Vehicule result = vehiculeRepository.findByImmat("AA-123-BB");
        //Then
        Assertions.assertThat(result).isEqualTo(v); 
    }
}
 Il est nécessaire de limiter le plus possible les tests de ce type car ils sont plus longs que les tests unitaires classiques à cause du temps d'initialisation du contexte Spring. On pourra notamment utiliser les Mocks.

Initialisation et Réinitialisation

 Lorsqu'on veut définir un contexte de test pour tous les tests d'une classe, on peut définir des méthodes permettant d'initialiser ou de réinitialiser ce contexte. 
public class ExempleTest {
  @BeforeClass //Junit 4
  public void setUp(){//Nom setUp arbitraire
    //Appelé une seule fois avant l'exécution des tests
  }
  @Before //Junit 4
  public void before(){//Nom before arbitraire
    //Appelé avant chaque test
  }
  @After //Junit 4
  public void after(){//Nom after arbitraire
    //Appelé après chaque test
  }
  @AfterClass // Junit 4
  public void tearDown(){//Nom tearDown arbitraire
    //Appelé une fois que tous les tests sont passés
  }
}
public class ExempleTest {
  @BeforeAll // Junit 5
  public void setUp(){//Nom setUp arbitraire
    //Appelé une seule fois avant l'exécution des tests
  }
  @BeforeEach // Junit 5
  public void before(){//Nom before arbitraire
    //Appelé avant chaque test
  }
  @AfterEach //Junit 5
  public void after(){//Nom after arbitraire
    //Appelé après chaque test
  }
  @AfterAll // Junit 5
  public void tearDown(){//Nom tearDown arbitraire
    //Appelé une fois que tous les tests sont passés
  }
}
 Ces méthodes peuvent être utiles pour vider la base de données ou définir un jeu de données communs à plusieurs tests. Une bonne pratique consiste à réinitialiser le contexte avant l'exécution du test (pour être sûr de ne pas récupérer de données de tests précédents).

Mock

 Pour conserver le principe de test unitaire, il est nécessaire de ne pas dépendre de l'implémentation d'éventuelles dépendances de la classe testée. Pour cela, on utilise les mocks avec Mockito (inclus avec Spring) et PowerMock. 
//Junit 4
@RunWith(MockitoJUnitRunner.class) 
public class VehiculeServiceTest {

  @InjectMocks
  private VehiculeService vehiculeService;
  @Mock
  private VehiculeRepository vehiculeRep;

  @Test(expected = EntityNotFoundException.class)
  public void testFindByImmatNotFound(){
    //Given
    Mockito.when(vehiculeRep.findByImmat("X")).
     thenReturn(null);
    //When Junit 4
    vehiculeService.findByImmat("X");
    //Then Exception
  }
}
//Junit 5
@ExtendWith(MockitoExtension.class) 
public class VehiculeServiceTest {
  //...
  @Test
  public void testFindByImmatNotFound(){
    //Given
    Mockito.when(vehiculeRep.findByImmat("X")).thenReturn(null);
    //When Junit 5
    EntityNotFoundException e = 
    Assertions.assertThrows(EntityNotFoundException.class, () -> 
     vehiculeService.findByImmat("X")
    );
    //Then Exception
    Assertions.assertEquals(e.getMessage(), "Véhicule X non trouvé!");
  }
}
 Ainsi, le vrai VehiculeRepository n'est jamais vraiment appelé, Mockito attrape l'appel et retourne un objet lui même.

Mock (2)

 Il est possible de vérifier des comportements sur les objets mockés. 
public class VehiculeServiceTest {
    @InjectMocks
    public VehiculeService vehiculeService;
    @Mock
    public VehiculeRepository vehiculeRepository;
    @Test
    public void testAddProprietaire(){
        //Given
        Vehicule vehicule = new Vehicule("AA-123-BB");
        Mockito.when(vehiculeRepository.findOne(Mockito.anyLong())).thenReturn(vehicule);
        Mockito.when(vehiculeRepository.save(Mockito.any(Vehicule.class))).then(Mockito.returnsFirstArg());
        //When addProprietaire(idVehicule, idProprietaire)
        vehiculeService.addProprietaire(1L, 2L);
        //Then
        ArgumentCaptor<Vehicule> vehiculeCaptor = ArgumentCaptor.forClass(Vehicule.class);
        Mockito.verify(vehiculeRepository, Mockito.times(1)).save(vehiculeCaptor.capture());
        Assertions.assertThat(vehiculeCaptor.getValue().getProprietaireId()).isEqualTo(2L);
    }
}
 On vérifie ici que le véhicule passé à la méthode save contient bien le proprietaireId et qu'il n'y a eu qu'un appel sans pour autant l'effectuer...

Les tests d'intégration

 Les tests d'intégration ont pour but de tester le bon fonctionnement de l'assemblage de modules (précédemment testés de manière unitaire) en un composant de plus haut niveau.
  • Un bon test d'intégration doit être...
  • indépendant
  • rapide
  • pertinent
 On ne doit pas re-tester les résultats produits par les sous-modules mais bien leur interaction !

Les tests d'intégration avec JUnit

 Les tests d'intégration sont des tests unitaires commes les autres si ce n'est qu'on n'introduit pas de mock. On appelle les vrais méthodes. Il y a donc généralement chargement du contexte Spring. 
@RunWith(SpringRunner.class) // Junit 4
@ExtendWith(SpringExtension.class) // Junit 5
@SpringBootTest
public class VehiculeServiceTest {

    @Autowired
    public VehiculeService vehiculeService;

    @Test
    public void test(){
        //Given avec de vraies données d'entrées

        //When avec appel des vraies méthodes de repository...
        vehiculeService.test();

        //Then avec de vraies vérifications...
    }
}
 Il est préférable de tester le plus possible en test unitaire et de limiter les tests d'intégration à la validation de l'assemblage de l'ensemble des modules.

Métriques

 Les métriques permettant de s'assurer de la fiabilité d'un logiciel sont les suivantes (liste non exhaustive) :
  • Résultats des tests unitaires et d'intégration (% de tests OK)
  • Couverture de code (%)
  • Couverture de code avec mutation (%)
  • Complexité (application, classe, méthode)
  • Lignes de code en commentaire
  Attention, viser systématiquement 100% de couverture de code n'est pas efficient ! Il est également nécessaire de bien paramétrer son gestionnaire de qualité afin d'éviter les faux positifs.

Capacité fonctionnelle

 La capacité fonctionnelle d'un logiciel est le degré de conformité du logiciel aux besoins du client.
  • Tests d'acceptation
  • Tests d'interface
  • Tests manuels
  • Métriques
 Il est important de connaître où est la valeur dans le logiciel pour le client, et dans la mesure du possible, commencer le développement des fonctionnalités ayant le plus de valeur.

Les tests d'acceptation

 Les tests d'acceptation vérifient la conformité fonctionnelle de l'application, c'est-à-dire sa capacité à produire les résultats décris dans les spécifications en fonction de scénarios fonctionnels.
  • Un bon test d'acceptation doit être...
  • non technique
  • uniquement fonctionnel
  • non équivoque
  • écrit par le client
  • écrit avant le développement
  • automatisé
 Les tests d'acceptation peuvent être considérés comme des spécifications. Lorsqu'ils sont fournis avant le début des développement, on parle d'ATDD ou Acceptance Test Driven Development.

Les tests d'interface

 Les tests d'interface permettent d'assurer le bon fonctionnement d'une interface graphique en simulant un comportement utilisateur. Du fait du caractère évolutif d'une interface graphique, ces tests sont souvent réalisés en fin de développement.
  • Un bon test d'interface doit être...
  • automatisé
  • déployé sur plusieurs plateformes
  • indispensable
  • ... Y a-t-il vraiment de bons tests d'interface ?
 Les tests d'interface sont souvent difficiles à développer, automatiser et maintenir. Ils doivent donc être mis en place que pour les éléments non testables autrement (positionnement d'éléments, affichage de message d'erreur, déclenchement des actions avec les éléments de l'interface...).

Les tests manuels

 Les tests manuels ne doivent pas pour autant être oubliés. Ils sont parfois plus pertinents que des tests d'interface difficilement maintenables car plus intelligents.
  • Un bon test manuel doit être...
  • automatisé
  • scénarisé
  • pas trop long...
 Les tests manuels sont évidemment coûteux mais peuvent avoir une vraie valeur ajoutée dans certains projets. Les scénarios testés sont généralement ceux déroulés lors de la phase de recette.

Métriques

 Il n'y a pas vraiment de métriques permettant d'assurer la capacité fonctionnelle. On peut néanmoins citer :
  • Résultats des tests d'acceptation (% de tests OK)
  • Résultats des tests d'interface (% de tests OK)
  • Résultats des tests manuels (% de tests OK)
  • Satisfaction client (largeur de sourire ?)
  Il est très difficile d'impliquer le client dans cette démarche car cela engage sa responsabilité et implique qu'il connaisse exactement son besoin...

Performances

 Les performances d'une application dépendent de son type. On ne mesure pas de la même manière les performances d'un site web que d'un batch d'intégration de fichier par exemple.
  • Tests de charge
  • Métriques
 Négliger les performances d'une application peut parfois coûter très cher lorsqu'il faut les améliorer une fois l'application développée et livrée...

Les tests de charge

 Il est important de commencer les tests de charge dès le début des développements car généralement, c'est au fur et à mesure des développements que les performances se déteriorent.
  • Un bon test de charge doit être...
  • pertinent
  • surveillé
  • complet
  • réaliste
 Le bon équilibre doit être trouvé car lorsqu'on améliore les performances d'un côté, c'est souvent au détriment d'un autre module.

Métriques

 Les métriques permettant de s'assurer des performances d'un logiciel sont les suivantes (liste non exhaustive) :
  • Web : Temps de chargement des pages (durée)
  • Web : Temps des recherches (durée)
  • Batch : Nombre de lectures/écritures (nombre par seconde)
  • Application : Temps de chargement
  • Temps d'exécution des tests
  • Evolution des tests de charge
 Il faut arriver à identifier les points faibles de l'application (souvent la base de données mais pas que...) car parfois la solution n'est pas logicielle (augmentation mémoire...).

Maintenabilité

 la maintenabilité correspond à la capacité d'une application livrée à être maintenue (maintenance corrective et évolutive) en maîtrisant les coûts, délais et son intégrité.
  • Journalisation
  • Documentation
  • Développements spécifiques
  • Livraisons
  • Métriques
 On s'occupe souvent de la maintenabilité en fin de projet, lorsque la phase d'exploitation approche. C'est en général moins coûteux de garder cet élément à l'esprit au fur et à mesure des développements.

Journalisation

 La journalisation permet de consigner les événements survenus dans l'application. Ces événements sont généralement consignés dans des fichiers (avec des rotations) ou dans une base de données.
  • Actions utilisateur (CRUD)
  • Compte-rendu de traitements automatiques
  • Informations sur le client (OS, navigateur...)
  • Seuils techniques ou fonctionnels atteints
  • Erreurs techniques ou fonctionnelles
  Il existe des outils pour exploiter de manière efficace ces logs qui peuvent être difficile à consulter.

Documentation

 La documentation du code est très importante pour assurer la lisibilité et la maintenabilité d'une application

Documentation projet

 Une documentation technique pour le projet est un bon moyen d'améliorer le temps de montée en compétences des nouveaux arrivants.
  • MkDocks
  • Jekyll
  • Sharepoint
  • Wiki
 C'est aussi un bon endroit pour regrouper les instructions de livraison, les caractéristiques des différents environnements...

Développements spécifiques

 Il est parfois utile de développer spécifiquement des modules pour faciliter la maintenabilité et l'exploitabilité d'une application.
  • Interface de monitoring de traitements
  • Pages web d'administration (CRUD)
  • Suivi des interventions
  • Suivi des montées de version
  • Automatisation de tâches d'exploitation
  • ...
 Ce genre de développement pouvant être coûteux, il faut s'assurer de leur ROI.

Livraisons

 Le processus de livraison consiste à packager une certain version d'un logiciel et à le déployer sur un environnement cible.
  • Un bon processus de livraison doit être...
  • automatisé
  • tolérant aux pannes
  • transparent
  • réversible
  • traçable
  • documenté
 Il existe des outils permettant de faciliter les livraisons et de réduire les risques.

Métriques

 Les métriques permettant de s'assurer de la maintenabilité d'un logiciel sont les suivantes (liste non exhaustive) :
  • Nombre de lignes de commentaires et % par rapport au code
  • Code smells (nombre)
  • Dette technique (durée)
  • Code dupliqué (% et nombre de lignes)
 Ces métriques sont à surveiller tout au long de la phase de développement.

Conclusion

 La qualité logicielle est à prendre en compte à toutes les étapes du développement d'un logiciel !
